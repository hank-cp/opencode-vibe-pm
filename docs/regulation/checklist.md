# 质量门控检查清单

## 用法

在流程的阶段转换点执行对应检查清单。所有项标记 `[x]` 后才能进入下一阶段。

---

## Phase 0→1 门控：计划就绪

**触发条件**: S6 审查计划前自检

### 完整性检查

- [ ] CHK001 设计文档已阅读理解，无歧义
- [ ] CHK002 S4 需求澄清访谈已完整执行，所有模糊点已澄清
- [ ] CHK003 执行计划覆盖了所有设计要点
- [ ] CHK004 列出了新增/修改的文件及路径
- [ ] CHK005 约束/风险/限制已评估并列明

### 可行性检查

- [ ] CHK006 关键依赖（AxioDB 等）已确认可用
- [ ] CHK007 技术方案可行性验证通过（如概念验证/POC）
- [ ] CHK008 没有引入不合理的复杂度
- [ ] CHK009 计划可增量交付，每个 Phase 有明确检查点

## Phase 3→4 门控：交付报告就绪

**触发条件**: 代码实现完成，测试通过，S10 检查清单自查

### 实现完整性

- [ ] CHK101 执行计划的任务已全部实现

- [ ] CHK108 无遗漏的 TODO/FIXME/HACK 标记（未注明原因的）

### 代码规范

- [ ] CHK110 日志输出使用英文

### 测试报告

```
测试通过率: {N}/{M} (N 通过 / M 总数)

```

## 常见违规模式（参考自查）

| 模式 | 说明 | 检查时机 |
|------|------|---------|
| 跳过 S4 直接实现 | 未澄清需求就开始设计 | S5 启动前 |
| Bug 修复中夹带重构 | 修复和无关的代码清理混在一起 | 代码审查 |
| "以后再说"的延迟决策 | 设计中未决定的点被无限期推迟 | S6 审查 |
| 跳过验证 | 自测就说"应该可以" | S10 自查 |
| 类型逃逸 | 用 `any`/`@ts-ignore` 绕过类型系统 | 所有检查点 |
