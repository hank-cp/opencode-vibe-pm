# 通用编码风格

> 本文件为未支持语言的通用编码规范。当项目使用的语言没有对应的特定规范文件时，以本文件为基准。

## 通用格式

- 统一使用 UTF-8 编码，换行符使用 LF
- 使用项目约定的格式化工具
- 缩进使用项目约定（建议 2 或 4 空格）
- 行长度建议不超过 100 字符
- 文件末尾保留一个空行

## 文件组织

### 命名

- 源文件命名使用项目约定（建议 `kebab-case` 或 `snake_case`）
- 目录名使用项目约定（建议 `kebab-case`）
- 测试文件命名按项目约定

### Import / 依赖引入

1. 标准库/内置模块
2. 第三方依赖
3. 项目内部模块

### 文件结构

- 每个文件尽量只有一个主要导出
- 辅助类型/函数使用命名导出
- 模块暴露出清晰的公共 API

## 命名规范

### 变量

- 使用项目约定（建议 `camelCase` 或 `snake_case`）
- 短作用域用短名字
- 长作用域用描述性名字
- 布尔变量用疑问词前缀
- 不使用匈牙利命名法

### 常量

- 使用项目约定（建议 `UPPER_SNAKE_CASE`）

### 函数

- 使用项目约定（建议 `camelCase` 或 `snake_case`）
- 动词开头

### 类型 / 接口 / 类

- 使用项目约定（建议 `PascalCase`）
- 接口/类型名不加前缀

### 枚举 / 联合类型

- 按项目约定

## 类型安全

- 避免使用动态类型/任意类型
- 优先使用类型注解/类型提示
- 显式错误处理，不吞掉异常

## 函数设计

### 参数

- 超过 3 个参数使用对象/结构体参数

### 返回值

- 优先返回具体类型
- 可能返回"空"的场景使用项目约定（如 `Result<T, E>`、`Optional<T>`、`(T, error)`）

## 控制流

- 优先使用提前返回，减少嵌套层级

## 异步处理

- 使用语言/框架推荐的异步模式

## 错误处理

- 所有错误必须显式处理
- 在系统边界进行输入校验
- 不在内部逻辑中添加"不可能发生"的错误处理

## 日志

- 使用项目统一的日志模块
- 日志消息使用英文
- 关键路径记录 info/debug 日志

## 注释与文档

- 代码注释使用英文
- 公共 API 添加文档注释
- 复杂逻辑添加解释性注释

## 占位代码

```
// TODO(username): 需要对接 API — 预计 v1.1
// FIXME(username): 并发场景下可能数据竞争 — 需要加锁
// HACK(username): 临时绕过限制 — v1.0 后替换
```
